<task type="auto">
  <name>Task 2: Implement transactional audit logging for news and events</name>
  <files>src/app/api/admin/news/[id]/route.ts, src/app/api/admin/events/[id]/route.ts</files>
  <action>
Refactor PATCH and DELETE handlers in both files to use Prisma interactive transactions.

Current pattern (double read - race condition risk):
```typescript
const oldNews = await prisma.news.findUnique({ where: { id } });  // Query 1
const news = await prisma.news.update({ where: { id }, data });   // Query 2
await logAction({ oldValues: oldNews, newValues: news, ... });    // Query 3
```

Replace with single transaction (atomic operation):
```typescript
const news = await prisma.$transaction(async (tx) => {
  const oldNews = await tx.news.findUnique({ where: { id } });
  if (!oldNews) throw new Error("News non trovata");

  const updatedNews = await tx.news.update({ where: { id }, data });

  await tx.auditLog.create({
    data: {
      userId: (session.user as any).id,
      action: "UPDATE",
      entityType: "news",
      entityId: id,
      oldValues: JSON.stringify(oldNews),
      newValues: JSON.stringify(updatedNews),
      ipAddress: request.headers.get("x-forwarded-for") || null,
      userAgent: request.headers.get("user-agent") || null,
    },
  });

  return updatedNews;
}, {
  maxWait: 5000,
  timeout: 10000,
});
```

Apply same pattern to:
1. news/[id]/route.ts - PATCH handler (lines 31-75)
2. news/[id]/route.ts - DELETE handler (lines 77-109)
3. events/[id]/route.ts - PATCH handler (lines 26-70)
4. events/[id]/route.ts - DELETE handler (lines 72-101)

Important:
- Remove the separate logAction() calls - inline the audit log creation within transaction
- Use tx.auditLog.create (NOT the external logAction helper) to stay within transaction
- Keep the logAction helper file intact (other code may use it outside transactions)
- JSON.stringify the oldValues and newValues (audit log schema stores them as strings)
- Add proper error handling with try-catch around the transaction
